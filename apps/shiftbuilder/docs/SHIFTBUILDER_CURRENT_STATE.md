- 更新日: 2026-07-14
- 対象ブランチ: `main`
- 目的: タスク再開時に長い会話や過去差分を読み直さず、現在の事実から作業を始める

## 現在のコード

- 現行ShiftBuilderは案件軸が実装済み。
- 月・エリア指定、案件×日付テーブル、KPI、候補者取得、アサイン、解除、楽観反映、失敗時ロールバックが存在する。
- 実データ由来の候補者と既存アサインを使う、読み取り専用の人員軸タブを公開済み。
- KPI直下の共通ハウトゥー、`H`単押し開閉、案件軸と人員軸の各行ゲージを公開済み。
- 人員軸セルは配置済み案件、未配置、同日重複を表示する。未配置を勤務可能とは表示しない。
- 人員の目標日数は実データ未確認のため、ゲージを「目標未設定」として表示する。
- `mock-data.js`は既存の開発用サンプルとして存在するが、本番機能の正本ではない。
- サンプルデータはURLへ`?demo=1`を明示した時だけ表示する。通常の認証失敗・API失敗では表示しない。
- 未ログイン時は実データを読み込まず、ログイン案内を表示するのが現在の基準挙動。

## 確定している仕様

- 人員軸は実際のShiftBuilderへ追加する。別のモックMVPは作らない。
- 案件軸と人員軸の共通ハウトゥーは、KPIクイックビュー直下へ置く。
- `H`キーを一度押すだけでハウトゥーを開閉する。
- 各行へ充足率ゲージを表示する。全体の仮ゲージは作らない。
- 人員軸は人員を行、日付を列、セル内は案件名と日別状態にする。
- 未配置を勤務可能と断定しない。
- 人員軸FIX後に、シフト確定・公開・配布機能を追加する。
- Googleカレンダー連携は、その後の工程とする。

詳細な設計意図と受け入れ条件は`260713_ShiftBuilder_人員軸タブ_Codex引継ぎ.md`を参照する。

## コードから確認済みのデータ

- 月次APIは`dates`と`cases`を返す前提になっている。
- 案件の日付セルは`required`、`assigned`、`candidates`等を扱う。
- 候補者APIは人員ID、表示名、コード、人員区分、契約区分、基準エリア等を扱う実装がある。
- 現在のアサインから、同一人員・同一日の重複は算出できる。

## 公開環境の実データで確認済み

2026-07-14に、ログイン済みのGitHub Pages版ShiftBuilderを読み取り専用で確認した。

- 2026年8月は31日、案件6行で描画され、KPIは`16 / 44`、不足`28`、充足率`36%`だった。
- 月次データ、候補者データ、現在ユーザーの取得は成功し、Consoleエラーは発生していなかった。
- アサイン済みセルでは、担当者名、完了状態、解除・変更操作へ進む既存UIを確認した。
- 未アサインセルの候補者には、表示名、アカウントコード、人員ID、人員区分、契約区分、基準エリアが表示される。
- 人員区分、契約区分、基準エリアには未設定の実データがあるため、欠損を前提に表示を設計する。
- 同日別案件の既存アサインを基にした競合警告が候補者単位で表示される。
- 人員軸は実データで人員10行、配置済み16セル、未配置294セルを描画し、全10行を「目標未設定」ゲージとして表示した。
- 人員軸から案件軸へ戻した後も、アサイン済みメンバー、候補者、解除、入れ替え候補の既存UIを確認できた。

個人名・認証情報・トークンは記録していない。上記の数値は確認時点の実データであり、固定値として実装しない。

## 環境上の注意

- 専用のlocalhost起動コマンドはリポジトリに定義されていない。
- 静的表示だけなら、リポジトリ直下で簡易HTTPサーバーを起動できる。
- 現在の確認用URLは`http://localhost:4174/apps/shiftbuilder/?demo=1`。デモ表示専用であり、実データ確認には使わない。
- Firebase Authenticationでは現在`127.0.0.1`が許可されておらず、localhostログインは`auth/unauthorized-domain`になる。
- この制約をコード変更や自動モック表示で隠さない。
