# ShiftBuilder「人員軸タブ」Codex実装引き継ぎ

- 作成日: 2026-07-13
- 対象: ShiftCore / ShiftBuilder
- 対象機能: 人員軸タブ
- 文書の目的: 次のCodexが、推測で既存仕様を壊さず、人員軸タブの設計・実装・検証を開始できる状態にする

> 重要: この文書は設計意図と作業ルールの引き継ぎであり、コードの現状を保証するものではない。実装開始時は、リポジトリに存在する最新コードを唯一の現状ソースとして読み直すこと。過去の会話、古い貼付コード、推定ファイル名を現状ソースとして扱わない。

## 1. 目的と背景

現在のShiftBuilderには「案件を主語にして、案件ごとの日付セルから人員を配置する」案件軸の画面がある。人員軸タブでは主語を反転し、「人員ごとの日別稼働状況を一覧し、その人員・日付に適した案件を配置する」画面を追加する。

人員軸は案件軸の単純な縦横入れ替えではない。利用者が判断したい内容が異なる。

| 軸 | 行 | 列 | セルの意味 | 主な判断 |
|---|---|---|---|---|
| 案件軸 | 案件 | 日付 | その案件・日付の配置状況 | この案件に誰を入れるか |
| 人員軸 | 人員 | 日付 | その人員・日付の稼働状況 | この人をどの案件へ入れるか |

人員軸で把握したい情報は、日別の未配置、アサイン済み案件、非稼働、休暇、同日重複、稼働日数の不足・超過などである。最終的には人員配置の偏りや異常を整えるための画面にする。

基本方針は次のとおり。

- 操作方法は案件軸を踏襲し、新しい操作学習を増やさない。
- 画面の主語だけを明確に反転させる。
- 色だけに頼らず、タブ、見出し、左固定列、セル内容で人員軸だと分かるようにする。
- 未配置を勤務可能と決めつけない。データ上区別できない状態は正直な文言で表示する。
- 既存のアサイン・解除・入れ替え・失敗時ロールバックを不用意に作り直さず、再利用可能性を確認してから接続する。

### この会話で追加確定した共通機能

次の2機能は人員軸だけの専用機能ではなく、**案件軸・人員軸の両方に入れる共通機能**とする。

- KPIクイックビュー直下の開閉式ハウトゥー
- 各行の充足率ゲージ

人員軸MVPの実装時に、案件軸へ後付けできない専用実装を作らない。見出し文言、集計対象、セル内の選択対象だけを軸ごとに差し替えられる共通部品または共通レンダリング方針にする。

性能上、ハウトゥーの開閉と充足率ゲージ自体は重い処理ではない。保留にするのは、月全体の表を不要に再描画する実装、または各セル描画中に同じ集計を繰り返す実装になった場合だけである。充足率はViewModel作成時または行単位の集計で一度だけ計算し、レンダラーは結果を表示する。

## 2. 案件軸と共通にする操作モデル

人員軸でも、案件軸と同じ直接操作を採用する。人員名を先にクリックしてから日付を選ぶ二段階操作にはしない。

### 必須操作

| 入力 | 人員軸の動作 |
|---|---|
| セルを直接クリック | その「人員 × 日付」セルを選択し、詳細を開く |
| `S` | 表の左上端にある最初の操作可能セルへフォーカスする |
| 矢印キー | 人員 × 日付セル間を上下左右に移動する |
| `Enter` | 選択セルの候補案件ポップオーバーを開く |
| `Esc` | ポップオーバーを閉じ、開く前のセルへフォーカスを戻す |
| `Enter` / `Space` | ポップオーバー内のフォーカス中ボタンを実行する |

案件軸ですでに実装されている可能性が高い補助挙動も、最新コードを確認したうえで共通化または再利用する。

- ポップオーバー内の矢印キー移動
- `Home` / `End` による先頭・末尾移動
- 初期フォーカスを最初の操作可能候補へ置く
- フォーカス移動時の `scrollIntoView()` による表示追従
- 閉じるボタン、外側クリック、`Esc` のいずれでも元セルへフォーカス復帰
- 入力欄・セレクト・テキストエリア・編集可能要素にフォーカス中は、`S`などのショートカットを奪わない

### 実装原則

- 人員軸専用として同じキーボード処理を複製する前に、案件軸から安全に共通化できるか確認する。
- 表セルはネイティブの`button`を優先し、クリック、`Enter`、`Space`の標準動作を利用する。
- フォーカスリングを消さない。マウス選択状態とキーボードフォーカス状態を視覚的に区別できること。
- ポップオーバーを再描画した後もフォーカスが迷子にならないこと。
- 表の横スクロールはホイールや二本指操作を前提とし、セルにフォーカスがある場合は矢印キーをセル移動に使用する。

## 3. 人員軸で変わる主語

操作体系は同じだが、表示内容と選択対象を反転する。

| 項目 | 案件軸 | 人員軸 |
|---|---|---|
| 左固定列 | 案件情報 | 人員情報 |
| セル | アサインされた人員名・人数・状態 | アサインされた案件名・状態 |
| セル詳細 | 候補人員 | 候補案件 |
| アサイン操作 | 案件へ人員を配置 | 人員を案件へ配置 |

人員軸では左固定列に人員を並べ、日付セル内には案件名の短縮表示を出す。セルを開いたら、その人員・日付に対する候補案件を選ぶ。

人員名自体はMVPでは情報表示とし、主操作にしない。人員名をクリック可能に見せると「人員を先に選ぶ必要がある」「行全体を選択する」「プロフィールが開く」と誤解されるためである。将来、人員詳細を追加する場合は別機能として明示する。

## 4. UI方針

### 4.1 案件軸から踏襲するもの

- ヘッダー、対象月、エリア、前月・翌月の配置
- KPIクイックビューのカードレイアウト
- 左固定列＋日付列の表構造
- 日付ヘッダー、土日表現、セル寸法、スクロール方法
- セル選択、ポップオーバー、フォーカス、キーボード操作
- 状態色の基本的な意味
- 保存中・失敗時・異常時のフィードバック方針

### 4.2 一目で人員軸だと分かる識別要素

次の4点を同時に使う。画面全体を別テーマ色にする必要はない。

1. 「案件軸」「人員軸」のタブを設け、人員軸タブに人型アイコンを付ける。
2. 画面見出しを「人員軸」、副見出しを「人員から案件を配置する」とする。
3. 左固定列に人員名と人員属性を表示する。
4. 日付セル内に案件名を表示する。

強い専用色で軸を区別すると、完了・警告・異常を示す状態色と競合する。人員軸の識別は色ではなく、主語と情報内容で行う。

### 4.3 左固定列の情報候補

MVPでは次を基本とする。実データのフィールド名は最新APIレスポンスを確認して決める。

- 氏名
- 社員コードまたはアカウントコード
- 人員区分・契約区分
- 拠点または基準エリア
- 当月のアサイン日数 / 目標日数または上限日数
- 行ごとの充足率ゲージ

情報が多すぎる場合は、氏名、コード、契約区分・拠点、`12 / 18日`、ゲージを優先する。

### 4.4 セル表示

- アサイン済み: 案件名または店舗名の短いラベル
- 複数件: `2件`など件数と異常色。必要に応じて詳細で全件表示
- 未配置: 原則として空白または「未配置」。勤務可能と確認できる場合だけ「空き」等を使用
- 休み・非稼働・対象外: それぞれデータ上判別できる範囲で表示
- 保存中: 既存案件軸と同じ保存中表現

案件名をセルに収めるための省略表示は許容するが、`title`、ツールチップまたは詳細で正式名称を確認できるようにする。

## 5. KPIクイックビュー候補

案件軸と同じカード配置を踏襲し、内容は人員軸向けにする。

### 推奨初期案

1. 対象人員数
2. 総稼働日数
3. 稼働不足人員数
4. 同日重複数

### 算出条件

- 対象人員数: 現在の月・エリア・フィルターに該当するユニーク人員数
- 総稼働日数: 同一人員・同一日を原則1日として数える。複数案件がある日は別途異常として扱う
- 稼働不足人員数: 目標日数が存在し、実稼働日数が目標未満の人員数
- 同日重複数: 同一人員・同一日に複数の有効アサインが存在する組み合わせ数

目標日数や勤務可能日のデータがない段階で「稼働不足」を推定しない。その場合はKPIを非表示または「算出不可」とし、暫定で対象人員数・総稼働日数・同日重複数など、根拠のある値だけを出す。

「未アサイン人員数」は意味が曖昧になりやすい。月内に1件も配置がない人を指すのか、配置可能日が残る人を指すのかを定義できない限り採用しない。

## 6. KPI直下の開閉式ハウトゥー

### 配置

`フィルター → KPIクイックビュー → 開閉式ハウトゥー → 人員 × 日付テーブル`の順に置く。表を操作する直前に案内を置くことで、必要な瞬間に説明へアクセスできるようにする。

### 閉じた状態の例

> 人員軸の使い方　セル選択 → 矢印移動 → Enterで候補案件を表示

### 開いた状態の内容

1. セルを選択: セルをクリック、または`S`で左上端セルへ移動
2. セルを移動: 矢印キーで上下左右へ移動
3. 詳細を開く: `Enter`で候補案件を表示
4. 案件を配置: 候補案件のアサイン操作を実行

### 開閉仕様

- 見出し全体をクリックして開閉できる。
- 見出しにはネイティブの`button`を使うことを優先する。
- `Tab`で見出しへフォーカスできる。
- `Enter`または`Space`で開閉できる。
- `aria-expanded="true|false"`を開閉状態と同期する。
- 必要に応じて`aria-controls`で説明領域を関連付ける。
- 展開・折りたたみを示す矢印アイコンを表示する。アイコンだけに状態を依存しない。
- 初回表示は開く。
- 2回目以降は`localStorage`で前回の開閉状態を記憶する。
- 保存キーは人員軸用と分かる固有名にし、案件軸の状態と衝突させない。
- `H`キーでの開閉はMVPでは実装しない。使用頻度の低い独自ショートカットを増やさないため。

## 7. 凡例と各行の充足率ゲージ

### 7.1 凡例

凡例は表の直前または表ヘッダー付近にコンパクトに置き、色だけでなくラベルも併記する。

- アサイン済み
- 対応必要
- 通常の未配置
- 休み / 非稼働 / 対象外
- 同日重複 / 上限超過 / 異常
- 保存中（既存UIに存在する場合）

### 7.2 行ごとの充足率ゲージ

各人員行の左固定列に、当月の稼働状況を示すゲージを付ける。

基本式:

```text
充足率 = 実稼働日数 ÷ 目標稼働日数 × 100
```

表示例:

```text
12 / 18日
[████████░░░░] 67%
```

条件:

- 同一人員・同一日の複数案件は実稼働日数として二重加算せず、別途同日重複として扱う。
- 目標値未設定の場合は0%にせず「目標未設定」と表示する。
- 100%超過を許容する表示にする場合も、ゲージだけでなく「超過」を文字で示す。
- 色だけで判断させず、日数と割合をテキストでも表示する。
- 目標が契約上限なのか期待稼働日数なのかをデータ定義で明確にする。

## 8. 色の意味

| 色 | 意味 | 使用例 |
|---|---|---|
| 緑 | 完了 / アサイン済み | 有効な案件が1件配置済み |
| 黄 | 対応必要 | 配置推奨、稼働不足、確認が必要な未配置 |
| 灰 | 休み / 非稼働 / 対象外 | 希望休、契約外、勤務対象外 |
| 赤・紫 | 異常 | 同日重複、上限超過、データ矛盾 |
| 白 | 通常の空き・未配置 | 対応必要とまでは判定していない未配置 |

重要: すべての未配置セルを黄色にしない。画面全体が黄色になると警告色の意味が薄れ、ユーザーが黄色を無視するようになる。黄色は「今見る価値がある」「対応すべき根拠がある」セルに限定し、通常の未配置は白にする。

また、勤務可否が不明なセルは「配置可能」と断定しない。白の「未配置」とし、データ上勤務可能と確認できた場合だけ配置推奨判定の対象にする。

## 9. 候補案件ポップオーバー

人員軸のセルを開いたときは、候補人員ではなく候補案件を表示する。

### 推奨並び順

1. 不足しており、対象人員を配置可能な案件
2. 拠点・エリアが近く、契約・スキル等の条件適合度が高い案件
3. 配置可能だが優先度が低い案件
4. すでに充足している案件
5. 配置不可・条件不一致の案件

同じグループ内の細かな並びは、次の順で検討する。

- 不足度
- エリア適合
- 契約・人員区分・スキル適合
- 案件優先度
- 開始時刻等の業務条件
- 安定した最後のキー（案件名やID）

### 表示・操作条件

- 操作可能な候補を上、無効候補を下へ置く。
- 無効候補には「配置不可」だけでなく理由を表示する。
- 正常候補、充足済み、条件不一致などのUI状態は、描画前にデータとして正規化してからレンダラーへ渡す。
- 描画後にDOMを走査して状態を付け足す方式は避ける。並び順・ラベル・disabled・警告文を一つの正規化処理で決める。
- 初期フォーカスは最初の操作可能な候補案件へ置く。操作可能候補がなければポップオーバー本体または閉じるボタンへ置く。
- 候補一覧の矢印移動、`Enter` / `Space`実行、`Esc`で閉じて元セル復帰は案件軸と同じにする。

既存アサインがあるセルで何を表示するかは、最新の案件軸仕様とAPI能力を確認する。MVPで既存の入れ替え・解除を安全に再利用できるなら接続し、できない場合はまず閲覧と新規アサインに限定する。新しい入れ替え仕様を推測して作らない。

## 11. データ上の前提と懸念

最大の懸念は「未アサイン」と「勤務可能な空き日」を区別できるかである。

```text
未アサイン ≠ 勤務可能
```

アサインがない理由には、希望休、所定休日、契約外、勤務不可、データ未入力、対象外などがあり得る。区別できないまま空き日として表示すると、画面の信頼性を損なう。

### 実装前に確認するデータ

- 人員の一意ID（原則`internal_user_id`）
- 氏名、社員コード、契約区分、人員区分、拠点・エリア
- 人員ごとの日別勤務可否、休暇、非稼働、対象外の状態
- 月間目標稼働日数または上限日数
- 現在の有効アサイン
- 案件の日付、必要数、エリア、状態
- 人員と案件の適合条件
- キャンセル・アーカイブ済みデータを除外する条件

### 表示上の安全策

- 勤務可否を判定できない場合は「空き」「配置可能」ではなく「未配置」と表示する。
- 配置推奨の黄色は、勤務可能と案件適合の根拠がある場合だけ使う。
- KPIとゲージは不足データから推定しない。算出不可または目標未設定を明示する。
- 表示名ではなく人員IDと日付をセルキーにする。
- 同名人員、氏名変更、案件名変更でも操作対象がずれないようにする。

## 12. UX・行動心理学上の理由

本機能の判断には、UI/UX設計と行動心理学の観点を必ず含める。

- **一貫性と学習転移**: 案件軸と同じ操作を使うことで、既に覚えた操作を人員軸へ転用でき、認知負荷を下げる。
- **直接操作**: 人員を先に選ばせず、目的のセルを直接クリックさせることで、余計な手順とモード誤認を減らす。
- **主語の視覚化**: タブ、見出し、左列の人員情報、セル内の案件名によって、色を解釈しなくても現在の軸が分かる。
- **段階的開示**: ハウトゥーは初回に開き、理解後は折りたたみ状態を記憶する。初心者支援と画面密度を両立する。
- **ジャストインタイム学習**: 操作説明を表の直前に置き、実際に操作する瞬間に参照できるようにする。
- **警告疲れの防止**: 黄色を全未配置に使わず、対応根拠のあるセルに限定する。希少な警告ほど注意を維持しやすい。
- **前注意的処理**: 色と進捗ゲージにより、一覧を逐一読まなくても不足・完了・異常の行を見つけやすくする。
- **色への非依存**: 数値、文字、ラベルを併記し、色覚差や意味の取り違えに対応する。
- **選択肢過多の抑制**: 操作可能・優先度の高い候補案件を上に置き、無効候補を理由付きで下へ送る。
- **空間記憶の維持**: ポップオーバーを閉じたとき元セルへ戻し、ユーザーが表内の現在地を見失わないようにする。
- **エラー予防**: 勤務可否不明を「空き」と断定せず、データが保証する範囲だけを操作可能にする。

## 13. 推奨実装順

### Step 0: 最新コードの棚卸し

最初にリポジトリを検索し、少なくとも次を確認する。ファイル名は過去情報から決め打ちせず、`rg`等で現物を探す。

- ShiftBuilderのHTML、CSS、エントリーポイント
- DOM参照、状態管理、API層
- 案件軸テーブルのレンダラー
- 詳細・プレビュー・アクションポップオーバーのレンダラー
- キーボードナビゲーション
- 候補者の正規化・並び替え
- アサイン、解除、入れ替え
- 楽観反映と保存失敗時ロールバック
- 月次データAPIと候補取得API

Step 0の成果物は、現状の責務マップ、再利用可能箇所、変更予定ファイル、未確認事項とする。ここではまだ大規模編集しない。

### Step 1: データ定義を固定

- 人員軸の行モデル
- 日別状態の列挙
- 実稼働日数、目標日数、同日重複の定義
- 未配置と非稼働の判別可否
- 人員軸候補案件に必要なAPIデータ

不足データがあれば、この段階でユーザーへ具体的にファイルまたはAPIレスポンスを依頼する。補完して進めない。

### Step 2: 軸状態とタブ

- 案件軸 / 人員軸タブ
- 現在軸の状態管理
- 見出し、副見出し、人型アイコン
- URLや再読み込み後に軸を維持するかを決める

既存案件軸を壊さない最小差分で追加する。

### Step 3: 人員軸ViewModel

案件中心の月次データから、人員中心の表示モデルへ変換する純粋関数を作る。レンダラー内で複雑な集計を行わない。

例:

```text
personRows[]
  person
  targetDays
  assignedDays
  fulfillmentRate
  dailyCells[date]
    state
    assignments[]
    caseLabels[]
    availability
    anomaly
```

### Step 4: 読み取り専用UI

- 人員 × 日付テーブル
- 左固定列
- セル状態と案件名
- KPI
- 凡例
- 充足率ゲージ
- 開閉式ハウトゥー

まず表示と集計を確認し、書き込み処理を接続しない状態でデータ解釈を検証する。

### Step 5: 共通キーボード操作

案件軸の既存操作を再利用または小さく共通化し、人員軸に接続する。案件軸の挙動を変える場合は回帰テストを必須とする。

### Step 6: 候補案件ポップオーバー

- 候補案件データの正規化
- 優先順位付け
- 無効理由
- 初期フォーカス
- 矢印、`Enter` / `Space`、`Esc`
- 元セルへのフォーカス復帰

### Step 7: 書き込み操作

既存APIと既存安全機構を確認し、新規アサインから接続する。解除・入れ替えは意味とパラメータが一致する場合のみ再利用する。

- 楽観反映
- 保存中表示
- API成功時の確定
- API失敗時のイジェクト / ロールバック
- 再取得後の選択セル・フォーカス維持
- サマリー、ゲージ、ポップオーバーの再計算

### Step 8: 回帰・性能確認

案件軸と人員軸の双方を確認する。必要なら最後にのみ描画最適化を行う。

## 14. 推奨モデル、Reasoning、作業境界、テスト条件

### 推奨モデルとReasoning

- 推奨: 作業環境で利用可能な最新のコーディング向けCodexモデル
- Reasoning: `High`相当
- 画面設計やデータ定義の監査、既存コードの責務整理、複数ファイル変更、回帰確認を含むため、速度優先の低推論設定より高推論設定を優先する。
- 固有のモデル名は製品更新で変わり得るため、この引き継ぎでは固定しない。実行時に利用可能なCodexモデルの中から最新・高性能のコーディング向けモデルを選ぶ。

高推論を推奨する理由:

- 案件軸と人員軸の共通化境界を誤ると、既存案件軸に回帰不具合を出しやすい。
- 未配置、勤務可能、非稼働の意味を混同すると業務判断を誤らせる。
- キーボード、フォーカス、ポップオーバー再描画、楽観更新が相互に影響する。
- UI/UX、アクセシビリティ、行動心理、API整合性を同時に評価する必要がある。

### 触ってはいけない箇所

根拠なく次を変更しない。

- 既存案件軸のアサイン・解除・入れ替えの業務ロジック
- 書き込み失敗時のイジェクト / ロールバック
- APIのリクエスト・レスポンス仕様
- 認証・権限判定
- Dashboardや他アプリへの導線
- OrderCase等のShiftBuilder外機能
- Firebaseや本番データ構造
- 既存の色トークンや共通CSSの意味
- キャンセル・アーカイブの既存運用
- ユーザーが作成した無関係な変更

共通化のために既存案件軸を編集する場合は、何を共通化し、案件軸の外部挙動が変わらないことをどう検証するかを先に示す。

### MVPテスト条件

#### 表示

- 案件軸と人員軸を切り替えても、月・エリア条件と画面状態が破綻しない。
- 人員軸では左固定列に人員、セルに案件名が表示される。
- 複数案件、未配置、非稼働、休み、異常、保存中が定義どおり表示される。
- KPIと各行ゲージが同じデータ定義で計算される。
- 目標未設定を0%として誤表示しない。
- 凡例と実際のセル色が一致する。

#### キーボード・フォーカス

- セル直クリックで対象セルを開ける。
- 入力欄外で`S`を押すと左上端セルへ移動する。
- 矢印キーで上下左右へ移動でき、端で不正なセルへ出ない。
- `Enter`で候補案件が開く。
- 初期フォーカスが最初の操作可能候補にある。
- 候補一覧をキーボードで移動・実行できる。
- `Esc`、閉じるボタン、外側クリック後に元セルへ戻る。
- フォーカス移動時に対象が見える位置へ追従する。
- month input、select、textarea等の入力中に`S`が奪われない。

#### ハウトゥー

- 初回は開いている。
- クリック、`Tab`＋`Enter`、`Tab`＋`Space`で開閉できる。
- `aria-expanded`が表示状態と同期する。
- 再読み込み後に前回状態が復元される。
- `H`キーは反応しない。

#### 候補案件と書き込み

- 操作可能な候補が上、無効候補が理由付きで下に並ぶ。
- 不足・エリア・適合条件による並びが仕様どおりで安定している。
- 新規アサイン成功後、セル、KPI、行ゲージ、候補一覧が更新される。
- 解除・入れ替えを実装した場合、それぞれ成功時に表示が整合する。
- API失敗時に楽観反映が戻り、誤ったアサインが画面に残らない。
- 同一人員・同一日の重複を検知し、正常な空きとして扱わない。
- 候補案件がない場合もポップオーバーを閉じられ、フォーカスが戻る。

#### 回帰

- 案件軸のセルクリック、`S`、矢印、`Enter`、`Esc`が従来どおり動く。
- 案件軸の候補人員の並び、アサイン、解除、入れ替え、失敗時ロールバックが壊れていない。
- 月変更、前月・翌月、エリア変更、自動読み込みが壊れていない。
- コンソールに構文エラー、null参照、未処理Promise rejectionがない。

## 15. GitHub直編集運用

このプロジェクトでは、GitHub上での直編集を前提とする場面がある。1工程ごとに次の順で進める。

```text
編集
↓
GitHub上で差分・括弧・カンマ・関数境界を目視確認
↓
commit
↓
GitHub Pages反映待ち
↓
Pagesで実動作確認
↓
Console確認
```

注意:

- 長大ファイルを安易に全文貼り替えさせない。現状コードの行数と変更量を確認し、部分置換の方が安全なら「置換前 / 置換後 / 追加位置」で指示する。
- JavaScriptオブジェクトへプロパティを追加するときは、直前行のカンマを含めて提示する。
- commit前の目視項目は重要点だけを短く示す。注意事項がある場合は詳しく説明する。
- Pages確認前に完了扱いしない。
- ユーザーの許可なく別機能の変更、ブランチ操作、PR作成、公開範囲変更を行わない。

## 16. 「できた！」の意味

このユーザーが「できた！」「確認OK」などと報告した場合、特段の訂正がない限り、次をすべて完了した意味として扱う。

- 修正済み
- commit済み
- GitHub Pagesへ反映済み
- Pages上で動作確認済み

同じ工程を再度未完了として案内したり、同じ変更をやり直させたりしない。次の工程へ進む前に、完了済み事項として引き継ぐ。

## 17. 最新コードを唯一の現状ソースにする

実装開始時は、必ず最新のリポジトリを読む。

- この文書に記載したファイル名や関数名は設計上の手がかりであり、存在を保証しない。
- 過去に貼られた`main.js`等をコピーして上書きしない。
- 会話上「実装済み」とされている機能も、最新コードで存在と挙動を確認する。
- 最新コードとこの文書が矛盾する場合は、コードの現状を事実として扱い、設計意図との違いをユーザーへ報告する。
- 不明な仕様を推測で補完しない。判断によって結果が変わる場合は、必要なファイル、APIレスポンス例、またはユーザー判断を具体的に求める。
- 既存の未コミット変更やユーザー変更を見つけた場合は、それを保持し、無関係な差分へ触れない。

## 18. Codexが実装指示を出す際の形式

### 対話方針

- 親しみがあり率直な、ナナチ風の話し方を希望している。
- ただし、安易に同意するイエスマンにはならない。違和感、重複、設計矛盾、運用事故の可能性は、根拠と代案を添えて建設的に指摘する。
- 確認項目は必要なものに絞る。一方で構文エラー、データ破壊、既存機能の回帰など重要な注意事項は省略しない。

各工程の実装指示は、最低限、次の4区分を明確に分ける。

### 現状

- 最新コードで確認できた事実
- 既に実装済みの機能
- 問題または不足
- 影響範囲

### 削除

- 削除するコード、DOM、スタイル、イベント
- 削除理由
- 参照元が残らないことの確認方法

削除がない場合は「なし」と明記する。

### 追加・変更

- 触るファイル
- 追加・置換位置
- 変更内容
- 期待する挙動
- 必要な場合だけ置換前 / 置換後コード

### まだ触らない

- 今工程では変更しない機能・ファイル
- 後工程へ送る項目
- 仕様未確定のため保留する項目

加えて、各工程の最後に次を短く付ける。

- 実装後の重要確認点
- 特有の注意事項
- Commit title
- Commit description

チェック項目を過剰に細分化しない。ただし、構文エラー、データ破壊、既存案件軸の回帰、フォーカス迷子、API失敗時の誤表示など、事故につながる注意事項は省略しない。

## 19. 初回作業でCodexが返すべきもの

人員軸実装を開始する最初の応答では、すぐに大規模コードを生成せず、最新コードを確認したうえで次を提示する。

1. 現在のShiftBuilder責務マップ
2. 案件軸から安全に再利用できる処理
3. 人員軸専用に必要な処理
4. データ上確認できたこと / できないこと
5. 最初の実装工程で触るファイル
6. 削除・追加・まだ触らない範囲
7. 最小のテスト条件

最初の実装単位は、原則として「軸タブと読み取り専用の人員軸ViewModel / 表示骨格」までに限定する。勤務可否や候補案件データが不明な状態で、配置推薦や書き込み動作まで一気に作らない。

## 20. 完了条件

人員軸タブのMVP完了は、単に表が表示された時点ではない。次を満たした時点とする。

- 案件軸と共通の操作モデルで人員 × 日付セルを操作できる。
- 人員軸であることがタブを見落としても判別できる。
- 左固定列、セル、KPI、凡例、ゲージの意味が一貫している。
- 未配置と非稼働をデータの保証範囲内で正しく扱っている。
- 候補案件の順序、無効理由、キーボード操作が機能する。
- 書き込みを実装した範囲では、成功時と失敗時の表示整合性が保たれる。
- 案件軸の既存操作とアサイン系動作に回帰がない。
- GitHub Pages上で確認し、Consoleに重大エラーがない。
- ユーザーが「できた！」「確認OK」と報告している。
